## はじめに

AIアシスタントと聞くと、多くの人はチャットウィンドウの裏にある大規模言語モデルを思い浮かべます。しかし、AIアシスタントを本当に「生きた存在」にするのは、その背後にあるランタイムアーキテクチャです。

OpenClawは独自の**4層アーキテクチャ**を採用し、コントロールプレーン・ゲートウェイ・エージェントランタイム・端末ノードを分離することで、軽量かつ強力なAIアシスタントランタイムを実現しています。

本記事では、OpenClawアーキテクチャの各層を深掘りし、なぜこの設計がTelegram・Discord・WhatsAppを同時に接続しつつ、Mac・iPhone・Androidデバイスまで制御できるのかを解説します。

## Gateway層：なぜシングルプロセス設計なのか

OpenClawアーキテクチャの最初の重要な設計判断は、Gatewayの**シングルプロセス設計**です。これは手抜きではなく、熟考の結果です。

### シングルプロセスの哲学

従来のマイクロサービスアーキテクチャでは、メッセージルーティング・認証・プラグイン管理を別々のサービスに分割します。しかし、個人AIアシスタントのユースケースでは、この分割による複雑性がメリットを大きく上回ります。

OpenClawのGatewayは、すべてのコア機能を1つのNode.jsプロセスに集約しています：

- 各Channelプラグインからのメッセージ受信とルーティング
- Nodesとの双方向WebSocket接続管理
- セッション状態の維持
- プラグインライフサイクル管理

シングルプロセスがもたらすメリット：**ゼロネットワークオーバーヘッドの内部呼び出し**、**シンプルなデプロイモデル**、そして**状態の自然な一貫性**。

### WebSocketプロトコル設計

GatewayとNodes間の通信にはWebSocketを採用し、メッセージは3つの型で構成されます：

```typescript
// req: リクエスト-レスポンス型
{ type: "req", id: "abc123", method: "camera_snap", params: { facing: "front" } }

// res: レスポンス
{ type: "res", id: "abc123", result: { image: "base64..." } }

// event: 一方向イベントプッシュ
{ type: "event", name: "location_update", data: { lat: 35.68, lng: 139.76 } }
```

同期呼び出しには`req/res`、非同期通知には`event`を使い、すべてのインタラクションパターンをカバーしています。

### セキュリティモデル：デバイスペアリングとToken認証

セキュリティはOpenClawの最低ラインです。各Nodeは初回接続時に**デバイスペアリング**フローを通過する必要があります：

1. Nodeがペアリングリクエストを送信、Gatewayがワンタイムペアリングコードを生成
2. ユーザーが信頼済みデバイスでペアリングを承認
3. ペアリング成功後、Gatewayが長期Tokenを発行
4. 以降の接続はToken認証で自動的に行われる

この仕組みはBluetoothペアリングに着想を得ています——初回の信頼確立後は、以降の接続が自動認証されます。

## Agent Runtime：コンテキスト組立とReActループ

Agent RuntimeはOpenClawアーキテクチャの「頭脳」です。ユーザーメッセージを完全なAIインタラクションに変換する責務を担います。

### コンテキスト組立メカニズム

AIが応答する前に、Agent Runtimeは**5つのソース**からコンテキストを組み立てます：

1. **システムプロンプト**：AIの身元、能力、行動境界を定義
2. **ワークスペースファイル**：SOUL.md、USER.md、AGENTS.mdなどの永続設定
3. **メモリファイル**：MEMORY.md（長期記憶）+ 日次メモリファイル
4. **セッション履歴**：現在の対話のコンテキストウィンドウ
5. **ツール結果**：先行ツール呼び出しの戻り値

```
System Prompt ──┐
Workspace Files ─┤
Memory Files ────┼──→ [Context Assembly] ──→ LLM API Call
Session History ─┤
Tool Results ────┘
```

この多元組立により、AIは各応答時に完全な「世界観」を持ちます。自分が誰で、ユーザーが誰で、何が起きたか、どんなツールが使えるかを理解しています。

コンテキストウィンドウの最適化戦略として、古いメッセージの自動圧縮・ツール出力の長尺時自動要約・メモリの関連性順ソートなどが実装されています。

### ReActループ

ReAct（Reasoning + Acting）はAgent Runtimeのコア実行モデルです。AIがツールを使う必要がある場合、以下のループに入ります：

```
ユーザーメッセージ → LLM思考 → ツール呼び出し決定 → ツール実行 → 結果取得 → LLM再思考 → ...
```

```typescript
// ReActループの疑似コード
while (true) {
  const response = await llm.chat(context);
  
  if (response.hasToolCalls) {
    const results = await executeTools(response.toolCalls);
    context.append(results);
    continue; // ループ継続
  }
  
  // ツール呼び出しなし → 最終応答を返す
  return response.text;
}
```

このループにより、AIは多段階推論が可能です。ウェブを検索→ファイルを読む→コマンドを実行→すべての情報を統合して回答、という流れが自然に実現します。

### Memory Flushメカニズム

Memory Flushは、OpenClawアーキテクチャで最も精巧な設計の1つです。対話が終了（無通信期間を検出）した後、Agent Runtimeが以下を実行します：

1. 今回の対話の重要情報を振り返る
2. 記憶すべきコンテンツを圧縮・抽出
3. `memory/YYYY-MM-DD.md`（日次メモリ）に書き込む
4. 必要に応じて`MEMORY.md`（長期記憶）を更新

これは人間が就寝前に一日を振り返り、重要なことを短期記憶から長期記憶に移すプロセスに似ています。AIアシスタントはセッションをまたいだ連続性を獲得し、「金魚の記憶」のチャットボットではなくなります。

## メモリシステム：Markdownをデータベースとして使う

OpenClawは大胆な選択をしました。**Markdownファイルをメモリの保存形式として採用**しています。SQLiteもRedisも使わない、純粋なテキストファイルです。

### 二層メモリ設計

| レイヤー | ファイル | 用途 | 人間の例え |
|----------|----------|------|-----------|
| 長期記憶 | `MEMORY.md` | ユーザー嗜好、重要な決定、永続的知識 | 長期記憶 |
| 日次メモリ | `memory/YYYY-MM-DD.md` | 当日の対話要約、イベント記録 | 日記 |

```markdown
<!-- MEMORY.md の例 -->
# Long-Term Memory

## User Preferences
- 日本語でのコミュニケーションを好む
- タイムゾーン：JST (UTC+9)
- 愛用ツール：VS Code, iTerm2

## Key Decisions
- 2026-03-01: ブログをAstroフレームワークに移行する決定
```

#### なぜMarkdownなのか？

- **可読性**：人間が直接メモリファイルを読み書きできる
- **バージョン管理**：Gitで自然にサポート、変更履歴を追跡可能
- **移植性**：テキストファイルなので依存関係ゼロ
- **AI親和性**：LLMはMarkdown形式の処理が得意

### ベクトル検索機能

メモリファイルが蓄積されると、単純なテキストマッチでは不十分になります。OpenClawは複数のembeddingプロバイダーをサポートするベクトル検索を統合しています：

```yaml
# ベクトル検索設定例
memory:
  embedding:
    provider: openai
    model: text-embedding-3-small
    dimensions: 1536
```

メモリファイルがチャンク分割されembeddingベクトルが生成され、ユーザーメッセージとのコサイン類似度で最も関連性の高いメモリ断片を見つけます。数百日分のメモリファイルがあっても、ミリ秒単位で関連情報を検索できます。

## プラグイン体系と拡張性

OpenClawの拡張性は**4つのプラグインスロット**の上に構築されています。

```
┌──────────────────────────────────────────┐
│            OpenClaw Plugin System         │
├──────────┬──────────┬────────┬───────────┤
│ Channel  │ Provider │ Memory │   Tool    │
├──────────┼──────────┼────────┼───────────┤
│ Telegram │ OpenAI   │ Local  │ Browser   │
│ Discord  │ Anthropic│ Vector │ Exec      │
│ WhatsApp │ Gemini   │ Custom │ Web Search│
│ Slack    │ Mistral  │        │ Camera    │
│ WeChat   │ Local LLM│        │ Custom    │
└──────────┴──────────┴────────┴───────────┘
```

- **Channelプラグイン**：メッセージの入出力を定義。各プラットフォーム用に1つずつ実装
- **Providerプラグイン**：異なるLLMプロバイダーと接続
- **Memoryプラグイン**：メモリの保存・検索方式を制御
- **Toolプラグイン**：AIの能力を拡張。ブラウザ操作・Shell実行・Web検索など

### Nodesのデバイス機能拡張

NodesはOpenClawアーキテクチャで最も革新的な部分です。物理デバイスの機能をAIアシスタントに公開します：

- **Camera**：写真撮影・動画クリップ録画（前面/背面カメラ）
- **Screen**：スクリーンショット・画面録画
- **Location**：デバイスの位置情報取得（精度制御可能）
- **Canvas**：デバイス上でUIコンテンツをレンダリング・表示

```typescript
// Node経由で写真を撮影する例
const photo = await nodes.camera_snap({
  node: "my-iphone",
  facing: "back",
  quality: 0.8
});
```

現在macOS・iOS・Androidの3大プラットフォームをサポートしています。

## まとめ：設計のトレードオフ

OpenClawアーキテクチャのコア理念は3つの言葉に集約されます：**シンプル・透明・拡張可能**。

- シングルプロセスGatewayは水平スケーリングを犠牲に、デプロイ・運用の極限のシンプルさを獲得
- Markdownメモリシステムはクエリ性能を犠牲に、人間可読性とバージョン管理を実現
- プラグイン化設計は即座の機能充実を犠牲に、無限の拡張可能性を提供

これらのトレードオフの背景にあるロジックは一貫しています。OpenClawは**個人AIアシスタント**シナリオに最適化されており、エンタープライズ級SaaSではありません。個人のメッセージ量に分散システムは不要ですが、AIが自分を本当に理解し、記憶し、デバイスを操作できることが重要です。
